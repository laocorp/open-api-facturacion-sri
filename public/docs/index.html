API de facturación electrónica SRI para Ecuador. Integración REST simple, multi-tenancy, firma XAdES-BES.

Inicio rápido

Emita su primer comprobante autorizado por el SRI en menos de 5 minutos. Solo necesita una API Key y enviar un JSON con los datos del comprobante.

1. Obtenga su API Key

Regístrese vía POST /onboarding o genere una nueva API Key desde el panel de administración. Recibirá un apiKey (prefijo público) y un apiSecret (mostrado una sola vez).

Autenticación

Todos los endpoints de emisión requieren autenticación. La API soporta dos mecanismos:

API Key (emisión SRI)

Incluya su API Key en el header X-Api-Key en cada petición:

# Header requerido en todos los endpoints de emisión
X-Api-Key: sk_live_a1b2c3d4e5f6
Importante: El apiSecret se muestra una sola vez al crear la API Key. Guárdelo de forma segura. No podrá recuperarlo después.

JWT (administración)

Los endpoints de administración (API Keys, webhooks, configuración de tenant) usan JWT. Obtenga su token vía POST /auth/login e inclúyalo como Bearer token:

Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

Multi-tenancy

Para operaciones multi-tenant, incluya el header X-Tenant-ID con el UUID del tenant destino:

X-Tenant-ID: a1b2c3d4-e5f6-7890-abcd-ef1234567890

Firma HMAC

Para mayor seguridad, puede firmar sus peticiones usando HMAC-SHA256. La firma permite verificar que la petición no fue alterada en tránsito y que proviene de un cliente autorizado.

Generar la firma

Construya el string to sign concatenando: HTTP_METHOD + "\n" + URI + "\n" + BODY. Luego genere el HMAC-SHA256 usando su apiSecret como clave y codifique en Base64.

const crypto = require('crypto');

function generateSignature(method, path, body, secret) {
  const stringToSign = method + '\n' + path + '\n' + JSON.stringify(body);
  return crypto
    .createHmac('sha256', secret)
    .update(stringToSign)
    .digest('base64');
}

// Uso
const apiSecret = 'tu_api_secret';
const signature = generateSignature('POST', '/sri/emitir/factura', payload, apiSecret);

// Enviar en el header
// X-API-Signature: <signature>
import hmac
import hashlib
import base64
import json

def generate_signature(method, path, body, secret):
    string_to_sign = method + '\n' + path + '\n' + json.dumps(body, sort_keys=True)
    digest = hmac.new(
        secret.encode('utf-8'),
        string_to_sign.encode('utf-8'),
        hashlib.sha256
    ).digest()
    return base64.b64encode(digest).decode()

# Uso
sig = generate_signature('POST', '/sri/emitir/factura', payload, 'tu_api_secret')
# X-API-Signature: <sig>
// PHP — HMAC signature
function generateSignature($method, $path, $body, $secret) {
    $stringToSign = $method . "\n" . $path . "\n" . json_encode($body);
    return base64_encode(
        hash_hmac('sha256', $stringToSign, $secret, true)
    );
}

// Header
// X-API-Signature: <signature>
Nota: La firma HMAC es opcional pero recomendada para entornos productivos. Si incluye X-API-Signature, la API verificará su validez antes de procesar la petición.

Emitir factura

POST /sri/emitir/factura

Endpoint principal para emisión de facturas electrónicas. Valida contra el esquema SRI, firma con XAdES-BES y envía al SRI en ambiente especificado.

Parámetros del emisor

CampoTipoDescripción
rucstringRUC del emisor (13 dígitos)
razonSocialstringRazón social del emisor
codDocstringCódigo del comprobante: 01 factura
establecimientostringCódigo de establecimiento (3 dígitos)
ptoEmisionstringCódigo de punto de emisión (3 dígitos)
dirMatrizstringDirección de la matriz
obligadoContabilidadstringSI o NO

Parámetros del comprador

CampoTipoDescripción
identificacionstringRUC, cédula o pasaporte
tipoIdentificacionstring04 RUC, 05 cédula, 06 pasaporte, 07 consumidor final
razonSocialstringRazón social o nombre del comprador
direccionstringDirección (opcional)
emailstringCorreo electrónico (opcional)

Ambientes

AmbienteValorDescripción
Pruebas1Comprobantes de prueba sin validez fiscal
Producción2Comprobantes reales autorizados por el SRI
Tip: Use ambiente 1 para desarrollo y pruebas. Cuando esté listo, cambie "ambiente": 2 para producción. No necesita cambiar nada más.

Guía de remisión

Documento electrónico que ampara el traslado de mercadería. Incluye origen, destino, transportista y los ítems transportados.

Códigos de error

La API utiliza códigos HTTP estándar para indicar el resultado de cada petición. Los errores del SRI se devuelven en el campo mensajes del response.

CódigoSignificadoSolución
400Datos inválidosRevise el formato de los campos. Consulte el esquema en Swagger.
401API Key inválida o faltanteVerifique el header X-Api-Key. Si usa HMAC, verifique la firma.
402Saldo insuficienteCompre un bundle vía POST /payphone/buy-bundle.
429Rate limit excedidoEspere y reintente. Considere actualizar su tier de rate limiting.
500Error interno del servidorEl comprobante pudo quedar encolado. Consulte el estado vía el log de webhooks.
Rate limits por tier: Basic 30 req/min, Professional 120 req/min, Enterprise 600 req/min, Unlimited sin límite.

Webhooks

Reciba notificaciones en tiempo real cuando sus comprobantes cambien de estado. Configure URLs de callback que la API llamará automáticamente ante cada evento.

Eventos disponibles

EventoDescripción
comprobante.autorizadoEl comprobante fue autorizado por el SRI
comprobante.rechazadoEl SRI rechazó el comprobante
comprobante.pendienteEl comprobante está en proceso de autorización
certificado.por_vencerEl certificado digital está próximo a vencer

Configuración

Administre sus webhooks vía POST /api-keys/:id/webhooks. Cada webhook incluye una firma HMAC en el header X-Webhook-Signature para verificar su autenticidad.

Payload del webhook: La API envía un POST con el evento, el comprobante y sus datos actualizados. Incluye el campo event y data con el objeto completo del comprobante.
